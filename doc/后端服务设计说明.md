# 新人上岗项目后端服务设计说明

## 1. 当前后端链路

当前服务端已经按下面这条链路完成：

```text
文件上传
  -> 文本解析
  -> 结构识别
  -> section 切分
  -> chunk 切分
  -> 知识元数据补全
  -> 本地知识库存储
  -> RAG 检索问答
  -> 小测题生成
  -> 综合测试题生成
  -> 判分与薄弱点反馈
```

对应实现文件：

- [server.js](/Users/mac/qianzhu Vault/project/新手上岗/server.js)

## 2. 当前模块划分

### 2.1 ingestion 文件接入层

负责：

- 接收上传文件
- 按文件类型解析文本
- 保存原始上传副本

当前支持：

- `txt`
- `md`
- `csv`
- `json`
- `html`
- `pdf`
- `docx`
- `pptx`
- `xlsx`

### 2.2 knowledge pipeline 知识加工层

负责：

- 文本清洗
- 标题抽取
- section 切分
- chunk 切分
- 关键词提取
- 知识类型识别

当前 chunk 会附带这些字段：

- `source`
- `sourceType`
- `sectionTitle`
- `chunkType`
- `keywords`
- `tokenCount`

### 2.3 retrieval + answering 检索问答层

负责：

- query 分词
- 对 chunk 做轻量打分
- 返回最相关片段
- 在有 `MIMO_API_KEY` 时把片段拼给 MiMo
- 在没有模型时走本地回答兜底

### 2.4 assessment 出题与判分层

负责：

- 基于章节生成 `quiz`
- 基于章节生成 `exam`
- 提交答案后计算得分
- 输出错题、薄弱章节和补训建议

## 3. 为什么这里建议使用缓存

建议使用，而且当前已经实现了两层缓存。

### 3.1 第一层：解析缓存

路径：

- `.cache/parsed-docs/*.json`

作用：

- 同一个文件重复上传时，不必重新做 `pdf/docx/pptx/xlsx` 解析
- 降低 CPU 和 IO 开销

### 3.2 第二层：项目缓存

路径：

- `.cache/projects/*.json`
- `.cache/latest-project.json`

作用：

- 同一批资料再次导入时，直接复用完整知识库结果
- 前端刷新后还能恢复最近一次项目状态

## 4. 当前为什么先不用数据库

这一版先用文件缓存是合理的，因为：

1. MVP 阶段部署简单
2. 方便本地验证整个训练链路
3. 没有引入额外运维成本
4. 当前数据量和并发量还不大

## 5. 什么时候该升级到数据库 / 向量库

当你出现下面任意一种情况，就应该升级：

1. 一个系统要服务多个企业租户
2. 单次知识库文件很多，chunk 数量超过几千
3. 问答需要更稳定的语义召回
4. 需要保存员工学习记录、考试记录、权限和组织数据
5. 需要增量更新，而不是每次整批重建

推荐升级方向：

- 业务数据：`PostgreSQL`
- 检索索引：`pgvector` 或独立向量库
- 对象存储：MinIO / OSS / S3
- 任务队列：BullMQ / Redis
